# 🚀 Vue 3 组合式 API 深度进击：从 Mixin 进化到 Composable

> **核心心法**：Mixin 是“侵入式”的内容合体，而 Composable（组合式函数）是“隔离式”的逻辑借用。

---

## 1. 变量作用域：Mixin 的“侵入” vs Composable 的“隔离”

这是两者最本质的区别。在 Composition API 中，**代码的执行**与**变量的获取**是分离的。

| 特性         | Mixin (Vue 2)                                                | Composable (Vue 3)                                               |
| :----------- | :----------------------------------------------------------- | :--------------------------------------------------------------- |
| **存在形式** | **合体式**。逻辑直接注入组件 `this`，组件与 Mixin 融为一体。 | **独立式**。逻辑运行在独立的函数闭包中，通过变量进行通信。       |
| **可见性**   | **全公开**。Mixin 里的所有属性对组件都是强制公开的。         | **按需暴露**。只有通过 `return` 出来的变量，组件解构后才能拿到。 |
| **安全性**   | **低**。极易发生命名冲突（覆盖），且变量来源如同“黑盒”。     | **高**。通过 ES6 解构重命名，完美避开冲突，来源清晰。            |

---

## 2. 深度拆解：当你执行 `const { x } = useA()` 时发生了什么？

很多初学者担心如果不解构所有变量，逻辑就不会运行。这是一个误区。

### ❶ 逻辑全量运行 (Execution)

只要你在 `setup` 中调用了 `useA()`，函数体内的**所有代码**都会被一行不落执行：

- **生命周期注册**：内部的 `onMounted` 等钩子会自动挂载到当前组件。
- **副作用监听**：内部的 `watch`、`setInterval` 等会立即启动。
- **内存开销**：内部定义的临时变量（即使没 return）也会存在于内存闭包中。

### ❷ 变量按需解构 (Selection)

解构仅仅是 JS 的一种**选取语法**。

- 你只解构 `loading`，并不代表 `count` 消失了，只是你的 `setup` 作用域没有拿到 `count` 的引用而已。
- **优势**：这保证了组件的简洁性，不会被 Hook 内部的 100 个中间变量污染。

---

## 3. 为什么 Composable 彻底终结了 Mixin？

### 解决“顺序黑盒”与“命名冲突”

- **Mixin 时代**：如果引入两个 Mixin 都有 `loading`，后面引入的会静默覆盖前面的。你只能靠控制数组顺序来祈祷不出错。
- **Composition 时代**：不再依赖顺序，而是依赖**手动重命名**。
  ```javascript
  // 完全掌控权在开发者手中
  const { loading: userLoading } = useUser();
  const { loading: listLoading } = useList();
  ```
