# vue查漏补缺

# 🌐 面试专项：前端路由 (Hash vs History)

## 1. 基础题：Hash 模式和 History 模式有什么区别？

建议从以下三个维度进行对比回答：

| 特性           | Hash 模式                               | History 模式                     |
| :------------- | :-------------------------------------- | :------------------------------- |
| **URL 表现**   | 带有 `#` (如 `example.com/#/user`)      | 正常路径 (如 `example.com/user`) |
| **底层 API**   | `window.onhashchange`                   | `history.pushState` / `popstate` |
| **后端配置**   | **不需要**。因为 `#` 后的内容不传给后端 | **必须配置**。需配合服务器重定向 |
| **SEO 友好性** | 较差                                    | 优秀                             |

---

## 2. 进阶题：为什么 History 模式刷新会 404，而 Hash 模式不会？

**核心原因：浏览器对 URL 片段的处理机制不同。**

- **Hash 模式**：浏览器在请求服务器时，会自动过滤掉 `#` 及其后面的内容。因此无论你在哪个路由刷新，服务器收到的始终是根路径的请求（如 `GET /`），从而能正确返回 `index.html`。
- **History 模式**：路径被视为真实的资源地址。当你刷新 `example.com/user` 时，浏览器会真实地向后端发起 `GET /user` 的请求。如果服务器物理磁盘上没有 `user` 这个文件夹或文件，且没有做特殊配置，就会返回 **404 Not Found**。

---

## 3. 实战题：Nginx 部署 History 模式除了 try_files，还要注意什么？

**回答重点：静态资源路径引用问题。**

- **现象**：配置了 `try_files` 后，首页能访问，但在二级路径（如 `/user/info`）刷新时，页面白屏，控制台报错 `Uncaught SyntaxError: Unexpected token '<'`。
- **原因**：这是因为浏览器试图去 `example.com/user/js/app.js` 寻找脚本，而 Nginx 找不到该资源，通过 `try_files` 返回了 `index.html`（HTML 第一行通常是 `<`），浏览器尝试将 HTML 当作 JS 解析，导致报错。
- **对策**：
  1.  **绝对路径**：确保 `index.html` 中引用的资源以 `/` 开头（如 `/js/app.js`）。
  2.  **配置 Base**：在 Webpack/Vite 中配置 `publicPath` 或 `base` 为 `/`。

---

## 4. 场景题：什么情况下必须使用 Hash 模式？

虽然 History 模式是目前的主流方案，但在以下场景中 **Hash 模式** 具有不可替代性：

1.  **无法控制服务器配置**：例如将静态页面部署在 GitHub Pages 或其他不支持 `rewrite` 的纯静态空间。
2.  **App 内嵌 H5 页面**：在混合开发（Hybrid）中，为了避免复杂的拦截逻辑，或者在某些老旧的 WebView 环境下，Hash 模式的兼容性和稳定性更高。
3.  **内网环境/工具类应用**：对 SEO 没有任何要求，且需要快速部署、即插即用，无需运维配合。

---

## 💡 补充：Nginx 核心配置参考

为了支持 History 模式，标准的 Nginx 配置应包含：

```nginx
server {
    listen       80;
    server_name  example.com;

    location / {
        root   /usr/share/nginx/html;
        index  index.html index.htm;
        # 核心逻辑：找不到物理文件时，回退到 index.html
        try_files $uri $uri/ /index.html;
    }
}
```

# 📋 面试专项：Vue 3 响应式监听器 (Watchers)

## 1. 基础题：`watch` vs `watchEffect` 有什么区别？

这是最基础的对比题，建议从以下三个维度回答：

| 特性           | `watch`                                                          | `watchEffect`                              |
| :------------- | :--------------------------------------------------------------- | :----------------------------------------- |
| **依赖追踪**   | **手动指定**。必须明确指出要监听的对象。                         | **自动追踪**。函数中用到什么就监听什么。   |
| **执行时机**   | **默认惰性**。仅在数据变化时触发（除非设置 `immediate: true`）。 | **立即执行**。创建时即运行一次以收集依赖。 |
| **旧值访问**   | **支持**。回调参数中包含 `oldValue`。                            | **不支持**。只能获取当前最新值。           |
| **代码简洁度** | 逻辑复杂或需精确控制时使用。                                     | 逻辑简单、多依赖并行时更简洁。             |

---

## 2. 深度题：`watchEffect` 在异步函数中有什么坑？

**核心痛点**：`watchEffect` 的依赖收集机制是**同步**的。

- **现象**：如果在 `watchEffect` 中使用了 `async/await`，那么**只有在第一个 `await` 之前的属性访问会被追踪到**。
- **原因**：一旦代码执行遇到 `await`，当前函数会暂停并放入微任务队列。此时 Vue 会认为该 Effect 的同步收集阶段已结束，将全局的“当前活跃 Effect”置为空。等 `await` 之后的内容执行时，响应式属性触发 Getter 也找不到订阅者了。
- **避坑指南**：如果异步之后需要用到某些响应式数据，请务必在 `await` 之前先读取（解构）它们。

---

## 3. 原理题：Vue 如何实现自动收集依赖？

面试官通常想考察你对“响应式系统”底层链路的理解：

1.  **全局标识**：Vue 内部维护一个全局变量（如 `activeEffect`），用来存放当前正在运行的监听函数。
2.  **触发读取**：执行 `watchEffect` 时，它会运行内部代码。代码中访问响应式变量（如 `count.value`）会触发该变量的 **Getter**。
3.  **依赖记录（Track）**：在 Getter 内部，该变量会检查 `activeEffect` 是否有值。如果有，就把这个 Effect 记录到自己的“依赖集合（Dep）”中。
4.  **变动通知（Trigger）**：当该变量被修改触发 **Setter** 时，它会遍历并执行刚才收集到的所有 Effect。

---

## 4. 细节题：如何在组件卸载前停止监听？

- **自动行为**：在组件的 `setup()` 或生命周期钩子中同步创建的监听器，会在组件卸载时自动停止。
- **手动停止**：如果监听器是在异步回调（如 `setTimeout`）中创建的，或者你想在特定逻辑下提前结束监听，可以捕获返回的 **Stop Handle**：

```javascript
// 执行 watch 或 watchEffect 会返回一个停止函数
const stop = watchEffect(() => {
  /* ... 监听逻辑 ... */
});

// 当满足某个条件时，手动停止监听
if (someCondition) {
  stop();
}
```
